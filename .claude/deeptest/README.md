# 브라우저 캡처 딥 테스트 가이드

CDP 브라우저 캡처가 OAuth 리다이렉트(Okta) 흐름을 녹화하는지 검증한다.

## 0. 자동 검증 결과 (완료 ✅)

실제 Chrome + CDP로 아래 시뮬레이터 흐름을 돌려, **오리진 전환 후의 XHR이 모두 캡처됨**을 확인:

```
GET 7001/        (서비스 홈)
GET 7002/authorize   ← 다른 오리진(Okta 역할)으로 바운스
GET 7002/login → 302
GET 7001/callback    ← 서비스로 복귀
GET 7001/api/items   ← 캡처된 XHR ✅
GET 7001/api/profile ← 캡처된 XHR ✅
```

리버스 프록시(7001 대상)였다면 7002 바운스와 그 뒤 세션을 놓쳤을 것. CDP는 전부 잡음.

## 1. 앱 수동 테스트 (로컬, Okta 불필요)

시뮬레이터로 앱의 '브라우저' 모드 전체 경로(전용 Chrome 기동→CDP→Mock 저장)를 확인.

```bash
# 1) 시뮬레이터 가동
node .claude/deeptest/oauth-redirect-sim.mjs
```

2) 앱에서:
   - 스펙은 `/api/items`, `/api/profile` operation이 있는 것으로 로드(또는 임의 스펙)
   - 상단 **프록시** 버튼 → 모달 → **브라우저** 탭
   - 시작 URL에 `http://127.0.0.1:7001` 입력 → **시작**
   - 전용 Chrome 창이 뜨면: **로그인** 클릭 → **계속** 클릭
   - 모달 녹화 목록에 `GET /api/items`, `GET /api/profile` 이 떠야 성공
   - **Mock으로** / **전체 Mock으로** 로 저장 확인
   - Chrome 창을 직접 닫으면 모달이 몇 초 내 **시작** 상태로 복귀하는지
   - 다시 시작 시 이전 세션 쿠키 유지(전용 프로필) 확인

## 2. 실제 Okta 테스트 (사내망)

1) 사내망 연결 + 필요 시 앱 설정에서 **SSL 검증 끄기**(사내 CA)
2) 브라우저 탭 시작 URL = 실제 서비스 진입 URL(로그인 전 페이지)
3) **시작** → 전용 Chrome에서 평소처럼 Okta 로그인 수행
4) 로그인 후 서비스 사용 → 호출되는 API(XHR/fetch)가 녹화 목록에 쌓이는지 확인
5) **전체 Mock으로** 저장 → baseURL에 경로 접두사가 있어도 `stripBasePath` 폴백으로 매칭되는지
6) 한 번 로그인하면 전용 프로필에 세션이 남아, 다음 녹화 때 재로그인 없이 바로 캡처되는지

### 주의/한계 (v1)
- **첫 탭만 캡처**: Okta가 별도 팝업창(window.open)으로 뜨면 그 창은 미캡처.
  대부분의 OIDC는 같은 탭 리다이렉트라 문제없음. 팝업 방식이면 로드맵의
  `Target.setAutoAttach` 확장 필요.
- HTML/JS/이미지 등은 의도적으로 제외(XHR/Fetch만 녹화).
- 바이너리/비-UTF8 응답 본문은 빈 문자열로 기록(녹화 항목 자체는 남음).
